# Architecture

## Stack

| Layer      | Tool                              | Purpose                                              |
| ---------- | --------------------------------- | ---------------------------------------------------- |
| Framework  | Next.js 14 (App Router)           | Frontend — pages, routing, server components         |
| UI         | Tailwind CSS + shadcn/ui          | Styling and component primitives                     |
| Backend    | Node.js / Express                 | API server — business logic, auth, DB access         |
| Database   | Supabase (PostgreSQL)             | Primary data store                                   |
| Storage    | Supabase Storage                  | Documents, lease PDFs, logos, tenant uploads         |
| Cache      | Redis                             | Session store, rate limiting, scheduled job locks    |
| Auth       | jsonwebtoken + bcryptjs           | Custom JWT — access token (httpOnly cookie) + refresh token (Supabase) |
| Email      | Resend                            | Receipts, rent reminders, owner statements, notifications |
| PDF        | PDFKit + pdf-lib                  | Receipts and statements (PDFKit), lease PDFs (pdf-lib) |
| WhatsApp   | wa.me URL links                   | Agent-initiated manual WhatsApp sends                |
| PWA        | next-pwa (Workbox)                | Offline support — payment recording syncs on reconnect |
| Language   | TypeScript strict                 | Throughout — frontend and backend                    |

## System Boundaries

| Layer              | Owns                                                                              |
| ------------------ | --------------------------------------------------------------------------------- |
| `app/`             | Pages, routing, layouts only. No business logic. No direct DB or API calls.       |
| `components/`      | UI only. No data fetching. No direct API calls. Receives data via props or hooks. |
| `hooks/`           | Data fetching via `api-client.ts`. State management. No UI rendering.             |
| `lib/api-client.ts`| All HTTP calls to the Express API. Single entry point — no raw `fetch` elsewhere. |
| `controllers/`     | Route handling only. Validate request, call service, return response.             |
| `services/`        | All business logic. DB access only here. Never in controllers or routes.          |
| `jobs/`            | Scheduled operations only. Use service layer — no direct DB calls in job files.   |

## Data Flows

### Flow 1 — User Interaction (Read)

```
Component renders
      ↓
useHook calls api-client.ts
      ↓
api-client sends GET to Express route (with access token cookie)
      ↓
auth.middleware verifies JWT
      ↓
Controller calls Service
      ↓
Service queries Supabase (always scoped to account_id)
      ↓
Redis cache checked first — DB queried only on miss
      ↓
Response returned → hook updates → component re-renders
```

### Flow 2 — User Interaction (Write / Mutation)

```
Agent submits form
      ↓
Hook calls api-client.ts POST/PATCH/DELETE
      ↓
auth.middleware → role.middleware
      ↓
Controller validates request body (zod)
      ↓
Service executes business logic + DB write
      ↓
Redis cache invalidated for affected keys
      ↓
Response → hook updates local state → UI reflects change
```

### Flow 3 — Scheduled Job (Rent Charge Generation)

```
Cron fires at 00:01 on configured due date
      ↓
Job acquires Redis lock (prevents duplicate runs)
      ↓
Job calls RentService.generateMonthlyCharges()
      ↓
Service queries all active tenancies due today
      ↓
For each tenancy: inserts charge record, updates tenancy status
      ↓
Triggers reminder schedule for each new charge
      ↓
Lock released
```

### Flow 4 — Payment Recorded (Online)

```
Agent records payment in UI
      ↓
PaymentService creates payment record
      ↓
PaymentService auto-generates receipt record + sequential number
      ↓
PDFKit generates receipt PDF
      ↓
PDF uploaded to Supabase Storage (receipts/{account_id}/{receipt_number}.pdf)
      ↓
Agent sends receipt: WhatsApp (wa.me link opened) or email (Resend)
      ↓
Communication log entry created against tenant
```

### Flow 5 — Payment Recorded (Offline → Sync)

```
Agent is offline — records payment in UI
      ↓
offline-queue.ts stores payment in IndexedDB
      ↓
Service worker detects reconnect
      ↓
Queue processes — POSTs each queued payment to Express API
      ↓
Same online flow from step 2 onwards
      ↓
UI refreshes — queue cleared
```

## Authentication

- Provider: Custom JWT (jsonwebtoken + bcryptjs)
- Methods: Email + password only
- Access token: short-lived (15 min), stored in httpOnly cookie (`access_token`)
- Refresh token: long-lived (30 days), hashed and stored in `refresh_tokens` table, delivered in httpOnly cookie (`refresh_token`)
- Rotation: new refresh token issued on every access token refresh — old one revoked
- Protected routes: all `/dashboard/*` and all Express API routes except auth endpoints
- Public routes: `/`, `/login`, `/register`, `/application/[token]`, and Express `POST /api/auth/login`, `POST /api/auth/register`, `GET /api/applications/public/[token]`
- On logout: refresh token revoked in DB, both cookies cleared
- RBAC: enforced in `role.middleware.ts` — admin can do everything, senior_agent most things, junior_agent read and record payments only

## JWT Pattern

```typescript
// Access token payload
interface AccessTokenPayload {
  sub: string;       // user.id
  accountId: string; // account.id
  role: 'admin' | 'senior_agent' | 'junior_agent';
  iat: number;
  exp: number;
}

// Signing (backend only — in AuthService)
const token = jwt.sign(payload, process.env.JWT_SECRET!, { expiresIn: '15m' });

// Verifying (in auth.middleware.ts)
const payload = jwt.verify(token, process.env.JWT_SECRET!) as AccessTokenPayload;
req.user = payload; // attached to every authenticated request
```

## Supabase Client Pattern

```typescript
// backend/src/db/client.ts
import { createClient } from '@supabase/supabase-js';

export const supabase = createClient(
  process.env.SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY! // service role — backend only, never expose to frontend
);

// All queries MUST include account_id filter
const { data, error } = await supabase
  .from('properties')
  .select('*')
  .eq('account_id', req.user.accountId); // always scoped
```

## Invariants

Rules the AI agent must never violate:

- Every DB query must include `.eq('account_id', req.user.accountId)` — no cross-account data leakage
- No DB calls in components, hooks, or Next.js pages — only in Express services
- No business logic in controllers — controllers validate input and call services only
- No raw fetch() in frontend — all HTTP calls go through `lib/api-client.ts`
- Never return `password_hash` in any API response — strip it at the service layer
- Never store secrets in frontend code or expose service role key to the client
- All currency amounts stored as numeric(12,2) — never as strings or floats
- Receipts are numbered sequentially per account — use a DB sequence, never application-level counters
- Trust transactions are append-only — never update or delete a trust_transaction row
- Tenancies are soft-closed (status = 'ended') — never deleted
